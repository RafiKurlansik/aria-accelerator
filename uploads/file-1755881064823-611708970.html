1. Designed with an open data architecture, promoting architectural interoperability. The software solution shall:
  1. Be built on a distributed microservices architecture with open, well-documented Representational State Transfer Application Programming Interfaces (API) to seamlessly interface with other systems and adapt to VA's evolving needs.
  2. Ensure that all data and any associated metadata, including, but not limited to, the following: business rules, data quality issues, transformations to the data performed within the platform is owned by and remains under VA control and can be exported in open, non-proprietary data formats via APIs.
  3. Enable data agnosticism that allows logic and analyses to be interoperable with all data types from any source at any scale.
  4. Enable quick and easy integration (using open standards and protocols) of new or legacy dataset structures from internal and external sources based on VA business stakeholder-defined requirements.
  5. Accommodate SQL Server Integration Service objects.
  6. Provide ability to independently develop new plugins, services, and extensions to meet new and unforeseen needs (e.g. COVID-19 response) and interfaces to external systems (e.g. Public Health Registries) without contractor or vendor support.
  7. Enable access to data stored within various types of databases internal and external to VA such as Open Database Connectivity, Java Database Connectivity and the Internet of Things.
  8. Provide the ability to export the following from the platform: the object data model, data quality metrics, data curation routines, descriptive statistics for inclusion in data monitoring and reporting functions, and analytical routines, for evaluation by Data Scientists and Data Analysts using open, standards-based APIs.
  9. Enable the export of raw and unstructured data, curated and refined data, and master or trusted or authoritative data for use in external applications and reporting solutions.
  10. Allow VA users to extract metadata documentation, workflows, business processes, and reports through open, standards-based APIs in order to promote interoperability.
  11. Enable the export of thousands of statistical indicators into various types of reporting and analytic systems.
  12. Enable batch ingestion of HL7, FHIR, and EHR data.
  13. Ability for end-users to interact with geospatial data (zoom, filter, mark maps, select, retrieve lists based on interacting with geospatial data).

F.   Maintain data origin and reproducibility of all integrated data sources. Every resource (e.g., dataset, analysis, report) shall retain its origin and metadata that can be traced back to the exact version of all upstream code and data dependencies. Both code and data shall be supported across different, coexisting branches according to industry-standard version-control workflows. The software solution shall:

  1. Provide the ability to create a dynamic data model, object-based search/discovery and analysis workflows, allow easy definition of objects, properties, and links that propagate from a source table, and provide intuitively designed ways to move between tabular and object-oriented interfaces and data analysis.
  2. Provide link analysis capabilities and allow users to create objects and add (or edit) attributes for any object to use in link analysis.
  3. Provide the users with interactive network diagrams that display associations between records of interest and other data defined objects.
  4. Provide the ability to discover metadata from imported data sets, data catalogs, and ETL code.
  5. Provide a unified mechanism of metadata management and data discovery through an intuitive user interface so analysts shall have a single point of reference while searching for data elements.
  6. Provide Data Managers and Data Engineers the ability to create, track, and manage data transfer processes.
  7. Ensure that all data written to the platform maintains its metadata, lineage, and provenance within an integrated data foundation (e.g., self-documenting code), and can be used alongside data native to the foundation.
  8. Provide the ability to clone and import/export data validation logic from the platform.
  9. Provide logic and model versioning, and metadata tracing for logic so users can understand the context for every piece of work created, including analyses and operational workflows.
  10. Support sequential sets of tables, or environments, on which processes (e.g. metadata management, data quality, and data curation) can operate and improve data.
  11. Support version control functionality to support quality rule implementation among concurrent users.
  12. Provide the ability to customize data lineage visualizations.
  13. Provide ability to import raw data, such as from spreadsheets or other sources for integration into data pipeline and reporting architecture.
  14. Provide configurable issue creation for datasets customizable to assign to specific groups with alerting feature.
  15. Provide tools to examine data health and allow users to receive notifications on data pipelines updates.
  16. Provide tools to interpret datasets time last built and lineage based up datedness within broader data pipeline.
  17. Enable automatic row counts for all datasets.
  18. Provide automated alerts when data row counts are not within an automatically calculated expected range.
  19. Provide automatic field-level statistics for all fields in datasets.
  20. Enable the ability to view build duration of each dataset and data performance metrics.

 

G.   The software solution shall allow VA individuals or VA data teams to create and use workflows. The software solution shall:

  1. Provide collaboration capabilities (e.g. shared workspace, reports) enabling teams comprised of technical and non-technical roles to work concurrently on the same data and merge analytical paths in one place.
  2. Allow users to use collaboration and knowledge management capabilities and allow analyst teams to collaborate on a project with all the information kept and shared in one location or repository (using GitHub or similar service), updated and worked on simultaneously.
  3. Allow analyst teams to work on the same analysis or assignment through file sharing capabilities.
  4. Provide work product and workspace sharing between users and allow users to control who has access to the collaboration.
  5. Provide the ability for users to create, track, adjudicate, and resolve data quality issues within the platform.
  6. Enable stewardship of data, including the ability to validate and remediate business rules and ensure data integrity.
  7. Allow for the creation of workflows in support of VA business and functionality.
  8. Allow for the creation, deletion and modification of in-system notifications or emails based on alerts or triggers.
  9. Allow access by teams external to VA to view and interact with reports generated using the software.
  10. Ability to track user view metrics of all reports and applications.
  11. Provide graphical user interfaces for pipeline visualization and management.
  12. Allow customizable build schedules with customizable alert features.
  13. Ability to create forms for user submitted input where input data is saved into datasets and available for rapid analysis.
  14. Provide version control of analytics including ability to roll-back to previous version.

 

H.   The software solution shall have the ability to provide proven granular security controls with the ability for data owners to easily control all downstream uses of the originating data and conform to VA security policies. The software solution shall:

  1. Meet FISMA High security controls.
  2. Meet security compliance framework policies including: FedRAMP High and Health Insurance Portability and Accountability Act (HIPAA).
  3. Provide security and privacy protection that allows multiple levels of access for different users based on Role Based Access Controls (RBAC), Classification Based Access Controls (CBAC), and Access Control Lists (ACL).
  4. Provide the ability to mask any data element (including part of a data element (e.g. first five digits of social security number) based on user privileges defined by RBAC, CBAC, and ACL.
  5. Propagate access control and data security through all data, data engineering transformation, data objects, data in IDE, all logic used in transformations, and all output data from IDE.
  6. Provide visibility of user access to and within the platform and retain an audit history of user access and actions for the duration of the contract.
  7. Maintain a history of user activity per session.
  8. Provide alerts based on abnormal usage patterns (e.g., attempts by data consumers to access sensitive data without proper privileges).
  9. Provide tools to examine user-specific permissions for specific datasets and data objects.
  10. Provide visualizations of permissions / data restrictions and markings for all datasets.
  11. Enforce Security Markings for VA systems. Mandatory controls known as “Security Markings” to establish strict eligibility criteria users must meet before gaining access to VA systems.
